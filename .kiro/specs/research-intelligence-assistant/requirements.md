# Requirements Document

## Introduction

The Research Intelligence and Benchmarking Assistant is a Python-based system that automatically generates professional research reports from a user-provided topic. Given a topic such as "AI-Based Debriefing in VR Medical Training" or "Haptic Feedback for Medical Simulation", the system searches multiple sources, collects and ranks relevant information, presents an intermediate review step, and produces a structured, evidence-backed report in Markdown format (v1 scope).

The system serves R&D, product, innovation, patent/IP, and technical leadership teams at organizations like VirtualiSurg, enabling them to rapidly understand the existing landscape — scientific, commercial, patent, and technical — around any research topic.

**V1 Scope Summary:**
1. User enters a topic
2. System searches for papers, patents, commercial solutions, and tools
3. System proposes benchmark metrics
4. User reviews and approves metrics and selected sources
5. System generates a Markdown report
6. System creates a basic Google Drive folder structure and index.md
7. System stores collected links and available documents

PDF, DOCX, Google Docs, Excel export, and advanced automation are deferred to later versions.

## Glossary

- **System**: The Research Intelligence and Benchmarking Assistant application
- **Topic**: The research subject provided by the user (e.g., "AI-Based Debriefing in VR Medical Training")
- **Workspace**: The structured Google Drive folder created per research topic
- **Report**: The final Markdown document (v1) generated by the System after user approval
- **Scientific_Paper**: A peer-reviewed or published academic article collected from scientific databases
- **Patent**: A patent document collected from patent databases (e.g., Google Patents, Lens.org)
- **Commercial_Solution**: A commercial product, service, or competitor offering relevant to the topic
- **Tool**: An integratable API, SDK, library, framework, or platform relevant to the topic
- **Standard**: An industry standard, guideline, regulatory reference, or best practice
- **Source_Item**: Any single collected item — paper, patent, commercial solution, tool, or standard
- **Benchmark_Metric**: A comparison criterion used to evaluate and score source items
- **Benchmark_Table**: A structured comparison table scoring all source items against benchmark metrics
- **Relevance_Score**: A numeric score (0.0–1.0) representing how relevant a source item is to the topic
- **Final_Score**: The average of all benchmark metric scores for a source item
- **Confidence_Level**: A qualitative rating (High, Medium, or Low) indicating certainty of extracted information
- **Validation_Step**: The intermediate user review phase before final report generation
- **Index_File**: The index.md file summarizing the workspace contents stored in Google Drive
- **Coverage_Value**: A numeric score for a single benchmark metric: Covered = 1.0, Partial = 0.5, Not Covered = 0.0
- **Search_Query**: The query string submitted to a data source during the search phase
- **LLM**: Large Language Model used for summarization, analysis, and metric generation
- **Drive_API**: Google Drive API used for workspace creation and file management
- **Semantic_Scholar**: A scientific paper API used as a data source
- **OpenAlex**: A scientific paper API used as a data source
- **Google_Patents**: A patent data source used by the System

## Requirements

### Requirement 1: Topic Input and Search Orchestration

**User Story:** As a researcher, I want to enter a research topic and have the System automatically search multiple sources, so that I can quickly collect all relevant information without manual searching.

#### Acceptance Criteria

1. THE System SHALL accept a research topic string as input from the user via the command-line interface.
2. WHEN a topic is submitted, THE System SHALL execute searches across scientific paper sources (Semantic_Scholar, OpenAlex, arXiv, PubMed, Crossref), patent sources (Google_Patents, Lens.org), commercial solution sources (web search), tool and API sources (web search and curated indexes), and standards sources (web search) in a single orchestrated run.
3. WHEN executing a search, THE System SHALL store each Search_Query string used, the source it was submitted to, and the timestamp of execution for traceability.
4. WHEN search results are returned, THE System SHALL extract for each Source_Item: title or name, source type, source URL, publication or access date, author or inventor or assignee or provider, and a short relevance explanation.
5. WHEN search results are returned, THE System SHALL assign a Confidence_Level of High, Medium, or Low to each Source_Item based on the quality and verifiability of the source.
6. IF a data source is unavailable or returns an error, THEN THE System SHALL log the failure, record "Source unavailable" for that source, and continue processing remaining sources without interrupting the overall run.
7. IF a data source enforces a rate limit, THEN THE System SHALL apply back-off and retry logic before marking the source as unavailable.

---

### Requirement 2: Result Ranking and Deduplication

**User Story:** As a researcher, I want collected results to be ranked by relevance and deduplicated, so that I receive a clean, prioritized list of the most useful findings.

#### Acceptance Criteria

1. THE System SHALL compute a Relevance_Score between 0.0 and 1.0 for each Source_Item based on semantic similarity to the topic and source quality signals.
2. THE System SHALL remove duplicate Source_Items by comparing titles, URLs, and DOI or patent numbers before presenting results to the user.
3. WHEN ranking Scientific_Papers, THE System SHALL prioritize peer-reviewed publications over preprints and grey literature.
4. WHEN ranking Patents, THE System SHALL prioritize active patents over expired or abandoned patents.
5. WHEN ranking Commercial_Solutions, THE System SHALL prioritize products directly related to the topic over tangentially related ones.
6. WHEN ranking Tools, THE System SHALL prioritize tools that can realistically be integrated into a software product based on available licensing, documentation quality, and active maintenance signals.
7. THE System SHALL include a short relevance explanation for each Source_Item stating why it was selected and how it relates to the topic.
8. THE System SHALL sort Source_Items within each category by Relevance_Score from highest to lowest before presenting them to the user.

---

### Requirement 3: Benchmark Metric Generation

**User Story:** As a researcher, I want the System to automatically suggest meaningful benchmark metrics for my topic, so that I have a structured comparison framework without needing to define criteria manually.

#### Acceptance Criteria

1. WHEN search results have been collected, THE System SHALL automatically generate a set of Benchmark_Metrics relevant to the topic using the LLM.
2. THE System SHALL present the proposed Benchmark_Metrics to the user before generating the final report.
3. WHEN computing the Final_Score for a Source_Item, THE System SHALL calculate it as the sum of all Coverage_Values divided by the number of Benchmark_Metrics.
4. WHEN assigning a Coverage_Value, THE System SHALL use 1.0 for Covered, 0.5 for Partial, and 0.0 for Not Covered.
5. THE System SHALL NOT generate the final Benchmark_Table until the user has approved or edited the proposed Benchmark_Metrics.

---

### Requirement 4: User Validation Step

**User Story:** As a researcher, I want to review and approve collected results and benchmark metrics before the final report is generated, so that I have full control over the report's content.

#### Acceptance Criteria

1. AFTER search and ranking are complete, THE System SHALL present a Validation_Step to the user displaying: proposed Benchmark_Metrics, list of collected Scientific_Papers, list of collected Patents, list of collected Commercial_Solutions, list of collected Tools, and list of collected Standards.
2. WHEN presenting the Validation_Step, THE System SHALL allow the user to approve proposed Benchmark_Metrics as-is.
3. WHEN presenting the Validation_Step, THE System SHALL allow the user to edit or remove individual Benchmark_Metrics.
4. WHEN presenting the Validation_Step, THE System SHALL allow the user to remove any Source_Item from the list.
5. WHEN presenting the Validation_Step, THE System SHALL allow the user to add missing Source_Items manually by providing a URL or reference.
6. WHEN presenting the Validation_Step, THE System SHALL allow the user to rename report sections.
7. THE System SHALL NOT begin final report generation until the user explicitly confirms the Validation_Step.
8. WHEN the user confirms the Validation_Step, THE System SHALL record the final approved list of Source_Items and Benchmark_Metrics before proceeding.

---

### Requirement 5: Evidence and Traceability

**User Story:** As a researcher, I want every claim in the report to be traceable to a verified source, so that the report is credible and defensible.

#### Acceptance Criteria

1. THE System SHALL assign a unique reference number to each Source_Item and include it in the report wherever the item is cited.
2. FOR every Source_Item cited in the report, THE System SHALL provide: source URL, access date, evidence excerpt or short source note, Confidence_Level, and reference number.
3. IF information about a Source_Item is missing or unverifiable, THEN THE System SHALL explicitly state "No public information found", "Not mentioned in available sources", or "Information could not be verified from public sources" rather than omitting the field or generating unsupported content.
4. THE System SHALL avoid generating claims that are not supported by at least one cited Source_Item.
5. THE System SHALL include a complete References section at the end of the report in IEEE citation format listing all cited sources with title, author or inventor or organization, year, URL, and access date.

---

### Requirement 6: Markdown Report Generation (V1)

**User Story:** As a researcher, I want the System to generate a complete, well-structured Markdown report after I approve the sources, so that I have a professional document ready for sharing and review.

#### Acceptance Criteria

1. WHEN the user confirms the Validation_Step, THE System SHALL generate a Markdown report containing all required sections as defined in requirements 7 through 13.
2. THE System SHALL use numbered headings throughout the report for all major sections and subsections.
3. THE System SHALL write all report text in clear, professional, and direct language that is accessible to non-technical stakeholders.
4. THE System SHALL NOT include unsupported claims, marketing language, or unverified statistics in the generated report.
5. WHEN report generation is complete, THE System SHALL save the Markdown file to the local filesystem and to the Final_Report subfolder in the Google Drive Workspace.
6. THE System SHALL confirm with the user before beginning report generation by displaying the approved source count and metric count and awaiting explicit confirmation.

---

### Requirement 7: Cover Page Section

**User Story:** As a researcher, I want the report to have a professional cover page with metadata, so that recipients can immediately identify the document's purpose, version, and origin.

#### Acceptance Criteria

1. THE Report SHALL include a cover page as the first section containing: report title, a "What is this document about?" subsection with a short purpose explanation, a Change Tracking table with columns Version, Date, Change, and Modified By, and document metadata including Author, Generated date, Topic, Version, and counts of papers, patents, commercial solutions, and tools analyzed.
2. THE Report SHALL display the report title at heading level 1 styled with the primary theme color #134f5c.
3. THE Report SHALL include the company logo reference in the cover page (path or placeholder when logo is not provided).

---

### Requirement 8: Table of Contents Section

**User Story:** As a researcher, I want an automatically generated table of contents, so that I can navigate quickly to any section of the report.

#### Acceptance Criteria

1. THE Report SHALL include an automatically generated Table of Contents linking to all major sections and subsections using Markdown anchor links.
2. THE System SHALL regenerate the Table of Contents automatically whenever the report structure changes.

---

### Requirement 9: Benchmarking and Gap Analysis Section

**User Story:** As a researcher, I want a benchmarking table comparing all collected items across meaningful metrics, so that I can quickly identify strengths, gaps, and innovation opportunities.

#### Acceptance Criteria

1. THE Report SHALL include a Benchmarking and Gap Analysis section containing a Benchmark_Table that lists all approved Source_Items as rows and all approved Benchmark_Metrics as columns.
2. THE System SHALL populate each cell in the Benchmark_Table with the Coverage_Value (1.0, 0.5, or 0.0) for that Source_Item and metric combination.
3. THE System SHALL compute and display the Final_Score for each Source_Item as the sum of its Coverage_Values divided by the number of Benchmark_Metrics, rounded to two decimal places.
4. THE System SHALL sort rows in the Benchmark_Table by Final_Score from highest to lowest.
5. THE Report SHALL include an Opportunity Analysis subsection identifying innovation gaps, competitive differentiation opportunities, unmet needs, white spaces, and potential innovation directions derived from the Benchmark_Table.
6. FOR every identified opportunity, THE Report SHALL provide: title, description, expected value, and estimated implementation complexity classified as Low, Medium, or High.

---

### Requirement 10: Executive Summary Section

**User Story:** As a researcher, I want a concise executive summary of all findings, so that stakeholders can grasp the key insights without reading the full report.

#### Acceptance Criteria

1. THE Report SHALL include an Executive Summary section containing: a concise one-to-one-and-a-half-line summary for each approved Source_Item, key findings, strategic insights, emerging trends, and recommended opportunities.
2. WHEN generating item summaries, THE System SHALL follow the format: "[Name] integrates/proposes/provides [main capabilities] to support [main purpose]." with a "read more" anchor link to the item's detailed section.
3. THE Report SHALL group Executive Summary item summaries by category in the order: Commercial Products, Scientific Papers, Patents, Integratable Tools, Standards.
4. THE Report SHALL sort items within each category by Relevance_Score from highest to lowest.
5. THE Report SHALL highlight the top five most relevant findings with a distinct visual marker in the Markdown output.

---

### Requirement 11: State-of-the-Art Detailed Sections

**User Story:** As a researcher, I want detailed structured sections for every collected item, so that I can dive deep into any paper, patent, product, or tool without leaving the report.

#### Acceptance Criteria

1. THE Report SHALL include a Commercial Products and Solutions subsection for each approved Commercial_Solution containing: company name as section title, overview, features table with columns Category, Feature Name, and Description, strengths and limitations, competitive positioning, and a summary table with Benchmark_Metrics scores.
2. THE Report SHALL include a Scientific Papers subsection for each approved Scientific_Paper containing: paper title as section title, context, method table with columns Step and Description, results, and a summary table with Benchmark_Metrics scores.
3. THE Report SHALL include a Patents subsection for each approved Patent containing: assignee name as section title, patent title, patent number, publication date, grant date, status, expiration date, invention explanation, figure reference or placeholder, claims summary, a Patent Lifecycle table with columns Tech Type, Country, Grant, Status, and Expiration, a Claims table, and a summary table with Benchmark_Metrics scores.
4. THE Report SHALL include an Integratable Tools subsection for each approved Tool containing: name, type, provider, source link, description, integration capabilities, key features, relevance, strengths, limitations, license and pricing, technical maturity, and a summary table with Benchmark_Metrics scores.
5. WHERE applicable, THE Report SHALL include a Standards, Guidelines, and Regulatory Considerations subsection listing relevant standards, compliance requirements, and their relevance to the topic.

---

### Requirement 12: References Section

**User Story:** As a researcher, I want a complete references section in IEEE format, so that all sources can be independently verified.

#### Acceptance Criteria

1. THE Report SHALL include a References section as the final section listing all cited Source_Items in IEEE citation format.
2. FOR every reference entry, THE Report SHALL include: reference number, title, author or inventor or organization, year or publication date, URL, and access date.
3. THE System SHALL number references sequentially in the order they first appear in the report body.

---

### Requirement 13: Google Drive Workspace Creation (V1)

**User Story:** As a researcher, I want the System to automatically create a structured Google Drive workspace for my research topic, so that all collected materials are organized and accessible.

#### Acceptance Criteria

1. WHEN the user confirms report generation, THE System SHALL ask the user for explicit confirmation before creating the Google Drive Workspace.
2. WHEN confirmed, THE System SHALL create a main Google Drive folder named "Research - [Topic] - YYYY-MM-DD" containing subfolders: Scientific Papers, Patents, Commercial Solutions, Integratable Tools, Standards and Regulations, and Final Report.
3. THE System SHALL create an index.md file in the root Workspace folder containing: topic, creation date, last update timestamp, folder structure summary, collected items counts by category, and links to subfolders.
4. THE System SHALL upload the generated Markdown report to the Final Report subfolder.
5. THE System SHALL store collected Source_Item links and any available downloaded documents in their respective subfolders.
6. IF the Drive_API returns an authentication error, THEN THE System SHALL display a clear error message instructing the user to configure Google Drive API credentials and halt the Workspace creation process without partial folder creation.
7. IF the Drive_API returns a permission error or quota error, THEN THE System SHALL log the error, notify the user, and continue saving outputs to the local filesystem as a fallback.

---

### Requirement 14: Research Workspace History

**User Story:** As a researcher, I want the System to maintain a history of all generated workspaces, so that I can revisit, update, or regenerate previous reports.

#### Acceptance Criteria

1. THE System SHALL maintain a local workspace history file recording for each completed run: topic, creation date, last update timestamp, report version, counts of papers, patents, commercial solutions, and tools analyzed, Google Drive Workspace link, and final report file path.
2. WHEN the user requests to update an existing report for a topic that already exists in the history, THE System SHALL ask for explicit user confirmation before overwriting the existing report.
3. THE System SHALL allow the user to list all previous workspaces from the command-line interface.

---

### Requirement 15: User Confirmation Gates

**User Story:** As a researcher, I want the System to ask for my confirmation before performing any significant irreversible action, so that I do not accidentally trigger unwanted operations.

#### Acceptance Criteria

1. THE System SHALL ask for explicit user confirmation before: creating the final report, downloading large files (greater than 10 MB), creating Google Drive folders, and updating an existing report.
2. WHEN the user declines a confirmation prompt, THE System SHALL cancel the pending action and return the user to the previous step without data loss.
3. THE System SHALL display a clear summary of the action to be taken in every confirmation prompt before awaiting user input.

---

### Requirement 16: Parser and Serializer Round-Trip (Internal Data Integrity)

**User Story:** As a developer, I want all data collected and transformed by the System to maintain integrity through serialization and deserialization cycles, so that no information is lost between processing steps.

#### Acceptance Criteria

1. THE System SHALL serialize all collected Source_Item records to a structured JSON format for intermediate storage.
2. FOR ALL valid Source_Item records, THE System SHALL guarantee that deserializing a serialized record produces an equivalent record (round-trip property: deserialize(serialize(x)) == x).
3. THE System SHALL validate each deserialized Source_Item record against the expected schema before using it in downstream processing.
4. IF a deserialized record fails schema validation, THEN THE System SHALL log the error, skip the invalid record, and continue processing remaining records.

## Out of Scope for V1

The following capabilities are documented for completeness but are explicitly deferred to future versions:

- DOCX report generation
- PDF report generation
- Google Docs native document creation
- Excel and CSV benchmark export
- Automatic PDF download of papers and patents
- Screenshot capture of commercial websites
- Multiple language support
- Web application or desktop application interface
- Citation verification before final export
- Predefined benchmark metric templates
- Advanced automation and scheduled re-runs

## Assumptions

- The user has permission to use the company logo in generated reports.
- The user has a valid Google account with access to Google Drive.
- Some commercial websites may block automated scraping; the System will store links rather than scraped content in those cases.
- Some sources may require API keys configured by the user prior to running the System.
- Some standards and regulatory documents may not be freely downloadable; the System will store references and links only.
- LLM-generated summaries may contain errors; the System encourages human review before business use of any report.
- Google Drive API credentials must be configured by the user before running workspace creation features.

---

## Risks

- Search APIs may return incomplete or outdated results for niche topics.
- Patent classification may be inaccurate due to complex legal language.
- LLM-generated summaries may hallucinate details not present in the source.
- Commercial product pages may contain marketing claims not independently verifiable.
- Google Drive API quota limits may interrupt workspace creation for large result sets.
- API rate limits from scientific paper sources may slow down collection for broad topics.
- Some PDFs may be behind paywalls and unavailable for download.
